# FIT — Freelance Interconnect

Cameroon-focused freelancing marketplace: clients publish jobs or order services, freelancers propose and deliver, payments are escrowed in FCFA via Mobile Money and released on client approval.

| App | Stack | Path |
|---|---|---|
| REST API + admin backend | Laravel 13 (PHP 8.4, MySQL, Sanctum) | `backend/` |
| Web dashboard (client/freelancer/admin) | Next.js 14 (React 18, TypeScript) | `web/` |
| Mobile app | Flutter 3.4x | `mobile/` |

API reference (auto-generated by Scramble): **`{API_URL}/docs/api`** — every endpoint with schemas, parameters and auth requirements. OpenAPI JSON at `/docs/api.json`.

## 1. Local development

### Backend (Laravel API)

Requirements: PHP 8.4, Composer, MySQL 8.

```bash
cd backend
composer install
cp .env.example .env          # then set DB_* credentials
php artisan key:generate
mysql -e "CREATE DATABASE fit" # or create the `fit` database another way
php artisan migrate --seed     # seeds catalog + demo data
php artisan serve              # http://localhost:8000
```

Demo logins (all password `password`): freelancer `diane@fit.cm`, client `mtn@fit.cm`, super admin `admin@fit.cm`, admin `support@fit.cm`.

Run tests: `php artisan test --compact` (Pest, 22 feature tests).

### Web dashboard

```bash
cd web
npm install
cp .env.local.example .env.local   # NEXT_PUBLIC_API_URL=http://localhost:8000/api/v1
npm run dev                        # http://localhost:3000  (/admin for the admin panel)
```

Production build: `npm run build && npm start`.

### Mobile app

```bash
cd mobile
flutter pub get
flutter run                        # iOS sim / macOS use localhost automatically
# Android emulator reaches the host via 10.0.2.2 automatically; real devices:
flutter run --dart-define=FIT_API_URL=http://<your-lan-ip>:8000/api/v1
```

Analyze/tests: `flutter analyze lib && flutter test`.

## 2. Environment variables

### Backend (`backend/.env`)

| Variable | Purpose |
|---|---|
| `DB_DATABASE` etc. | MySQL connection (database name: `fit`) |
| `FRONTEND_URL` | Comma-separated browser origins allowed by CORS (e.g. `https://app.fit.africa`) |
| `SANCTUM_TOKEN_EXPIRATION_MINUTES` | API token lifetime (default 10080 = 7 days) |
| `PAYMENT_WEBHOOK_SECRET` | Shared secret for the Mobile Money webhook signature — **required in production** |
| `API_VERSION` | Version string shown in `/docs/api` |
| `QUEUE_CONNECTION` | `database` by default — run `php artisan queue:work` in production |

### Web (`web/.env.local`)

| Variable | Purpose |
|---|---|
| `NEXT_PUBLIC_API_URL` | Base URL of the API, e.g. `https://api.fit.africa/api/v1` |

### Mobile (compile-time)

| Define | Purpose |
|---|---|
| `FIT_API_URL` | `--dart-define=FIT_API_URL=https://api.fit.africa/api/v1` at build time |

No secrets live in any repository file; tokens are stored in `localStorage` (web) and `SharedPreferences` (mobile).

## 3. Payments: sandbox vs production

Payment endpoints (`POST /orders/{id}/pay`, `/milestones/{id}/pay`, `/connect-packs/{id}/purchase`) create a **pending** payment intent and return a `FIT-PAY-…` reference.

- **Sandbox / development**: settle it with `POST /api/v1/payments/{reference}/confirm-sandbox` (both frontends do this automatically). This endpoint returns 404 in production.
- **Production**: the Mobile Money aggregator must call `POST /api/v1/webhooks/payments` with body `{"reference", "status": "successful|failed", "provider_ref"}` and header `X-Webhook-Signature: HMAC-SHA256(raw body, PAYMENT_WEBHOOK_SECRET)`. Unsigned or mis-signed calls are rejected with 401; the webhook returns 503 if the secret is not configured.

**Unresolved external dependency:** a Mobile Money aggregator contract (MTN MoMo / Orange Money API credentials) is required before real money can move. Until then the platform runs end-to-end on the sandbox flow.

## 4. Deploying to production

### Backend

1. Provision PHP 8.4 + MySQL 8 (VPS with nginx+php-fpm, or [Laravel Cloud](https://cloud.laravel.com)/Forge).
2. `composer install --no-dev --optimize-autoloader`
3. Set `.env`: `APP_ENV=production`, `APP_DEBUG=false`, `APP_URL`, DB credentials, `FRONTEND_URL`, `PAYMENT_WEBHOOK_SECRET` (generate with `openssl rand -hex 32`).
4. `php artisan migrate --force` and, first deploy only, seed the catalog (skip `DemoSeeder` in production):
   `php artisan db:seed --class=GeoSeeder --force && php artisan db:seed --class=CatalogSeeder --force && php artisan db:seed --class=PlatformSeeder --force`
5. `php artisan config:cache && php artisan route:cache && php artisan storage:link`
6. Run a queue worker (`php artisan queue:work --tries=3`, via supervisor/systemd).
7. Point the aggregator's webhook at `https://api…/api/v1/webhooks/payments`.
8. Health probe for the load balancer: `GET /api/v1/health` (checks DB) or `GET /up`.
9. Schedule DB backups (e.g. `mysqldump` cron) before launch — spec checklist requirement.

Rate limiting: 120 req/min per user/IP globally, stricter limits on auth endpoints. API docs at `/docs/api` are public by default only in `local`; enable in production by defining the `viewApiDocs` gate if desired.

### Web

Standard Next.js deploy (Vercel or node server):

```bash
NEXT_PUBLIC_API_URL=https://api.fit.africa/api/v1 npm run build
npm start   # or push to Vercel with the env var set
```

### Mobile

```bash
flutter build apk  --release --dart-define=FIT_API_URL=https://api.fit.africa/api/v1
flutter build ipa  --release --dart-define=FIT_API_URL=https://api.fit.africa/api/v1
```

## 5. What is intentionally not wired yet

- **Real Mobile Money settlement** — needs an aggregator contract (see §3). Sandbox flow covers the full escrow lifecycle meanwhile.
- **OTP delivery** — codes are generated and verified server-side but logged (`storage/logs`) instead of sent; plug an SMS/WhatsApp provider into `OtpController::send`. In debug mode the API returns `debug_code`.
- **Admin "Editorial Blog" tab (web)** — front-end only; no blog module exists in the backend/spec.
- **Push notifications (FCM)** — in-app notification center is live; device push is not.
- **File attachments from the mobile app** — the API accepts uploads (portfolio, deliveries, chat); the Flutter screens currently send text/links only.
